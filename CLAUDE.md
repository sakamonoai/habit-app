# 習慣化アプリ — プロジェクトルール

## プロダクト概要

日本版チャレンジャーズ型習慣化アプリ。
「固定デポジット × 証拠写真投稿 × 少人数グループ × スポンサー報酬」モデル。

**一言説明**: 仲間と一緒にチャレンジして、達成したらお金が戻ってくる。スポンサーからのご褒美もある。

## 技術スタック

| 領域 | 技術 |
|------|------|
| フロントエンド | Next.js (App Router) + TypeScript |
| スタイリング | Tailwind CSS |
| デプロイ | Vercel（git push → 自動デプロイ） |
| DB・認証・ストレージ | Supabase（PostgreSQL + Auth + Storage） |
| 決済 | Stripe（参加費受付・返金処理） |
| プッシュ通知 | Web Push / PWA |
| アプリ形式 | PWA（ネイティブアプリ不要） |

## 【最重要】絶対に破ってはいけないルール

- **未達者のデポジットを達成者に分配するコードは絶対に書かない**
- 決済情報（カード番号等）をDBや変数に直接保存しない。Stripeに委託する
- 本番DBを直接操作しない。マイグレーションファイル経由のみ
- ユーザーの証拠写真を無断でスポンサーに渡すコードを書かない
- 本番環境の環境変数を `.env` ファイルにコミットしない

## やってはいけないこと（地雷マップ）

- SSHで本番サーバーのファイルを直接編集しない（git管理から外れる）
- `git push --force` を main ブランチに実行しない
- Stripeの Webhook 署名検証をスキップしない
- 達成率計算ロジックを変更するときは必ずテストを書いてから
- 頼んでいない箇所のリファクタリングをしない。頼んだことだけやる

## 設計原則

### お金の流れ
```
参加費（例:1,000円）+ システム利用料（例:100円）
  ↓
85%以上達成 → 参加費を返金
100%達成   → スポンサー景品・特典
85%未満    → 参加費は返金なし（運営収益）
※ 未達者の金を達成者に分配しない
```

### 達成率計算
```
達成率 = 投稿した日数 ÷ チャレンジ期間の日数 × 100
```

### PWA方針
- App Storeの審査待ちをなくすため、ネイティブアプリではなくPWAで提供
- カメラ: `getUserMedia` / `<input type="file" capture>`
- プッシュ通知: Web Push API + Service Worker
- ホーム画面追加を促すonboarding UIを用意する

## ファイル・フォルダ構成

```
/app                    # Next.js App Router
  /(auth)               # 認証ページ（ログイン・登録）
  /(app)                # アプリ本体（認証済みユーザーのみ）
    /challenges         # チャレンジ一覧
    /challenges/[id]    # チャレンジ詳細
    /group/[id]         # グループタイムライン
    /dashboard          # 個人ダッシュボード
  /api                  # APIルート
    /stripe             # 決済Webhook
    /checkin            # チェックイン投稿
/components             # 共通コンポーネント
/lib                    # Supabase client, Stripe, utilities
/types                  # TypeScript型定義
```

## MVP スコープ（これだけ作る）

含める機能：
- ユーザー登録・ログイン（Supabase Auth）
- チャレンジ一覧・詳細表示
- 参加申込み・決済（Stripe）
- 証拠写真投稿（カメラ撮影）
- グループタイムライン（投稿閲覧・リアクション）
- 達成率計算・返金処理
- プッシュ通知（チェックイン催促）
- PWA対応（manifest + Service Worker）

含めない機能（後フェーズ）：
- スポンサー向け管理画面
- 独自チャレンジ作成（スポンサー）
- MVP選定機能
- 詳細分析・レポート

## デバッグの手順

1. Supabaseのダッシュボードでクエリログを確認
2. Stripeのダッシュボードでwebhookの受信状況を確認
3. Vercelのファンクションログでエラーを確認
4. ローカルで `npm run dev` で再現してから修正

## 注意：法律面

このプロジェクトは賭博罪・景品表示法・資金決済法の確認が必要。
現時点では「安全設計（山分けなし・スポンサー報酬分離）」で進めているが、
法律面で設計変更が必要になる可能性がある。
決済・返金フローは弁護士確認後に最終確定する。
